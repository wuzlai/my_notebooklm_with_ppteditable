**第1页**
**标题**: 简单场景：AI 提效显著但仍需人工干预
**副标题**: 案例A — 销售订单报表查询（低复杂度验证）
**内容要点**:
1. **效率对比**：Claude Code 耗时仅 30 分钟，较手写开发提效 50%，而 Copilot 无明显提升。
2. **生成质量**：两款 AI 初次生成均无法直接运行，均存在地址取数逻辑或字段虚构问题。
3. **运行风险**：Copilot 生成代码导致系统崩溃（Short Dump），Claude Code 运行相对平稳。
4. **性能瓶颈**：AI 在 SQL 优化方面表现欠缺，存在大表关联不当及未去重等性能隐患。
**视觉建议**: 左右对比条形图，展示耗时差异；使用“闪电”图标代表 Claude Code 的速度优势。

**第2页**
**标题**: 中等难度：数据字典“幻觉”导致代码不可用
**副标题**: 案例B — 采购配额维护（中等复杂度验证）
**内容要点**:
1. **业务理解偏差**：Copilot 完全混淆采购配额与货源概念，导致核心数据模型错误。
2. **虚构字段危机**：Claude Code 虚构字段占比高达 50%，直接引发 21 个连锁语法错误。
3. **接口规范缺失**：AI 无法正确识别 SAP 函数模块（FM）接口仅接受 DDIC 类型的规则。
4. **修复成本极高**：由于虚构字段与逻辑错误交织，AI 生成代码的修复成本远超重写成本。
**视觉建议**: 使用“警告”图标标注虚构字段占比；对比漏斗图展示从需求到可用代码的流失。

**第3页**
**标题**: 高复杂度：错误“修复-爆炸”模式导致效率反降
**副标题**: 案例C — 跨工厂 STO 报表（高复杂度验证）
**内容要点**:
1. **效率负增长**：Claude Code 耗时 8 人天，比手写开发（5 人天）多出 60% 的工作量。
2. **错误爆炸模式**：语法错误呈现 3→2→9 的非线性增长，修正旧错误会激发更多新连锁错误。
3. **严重虚构现象**：累计虚构 10 余项数据类型与表字段，AI 无法处理 18 张表以上的复杂关联。
4. **开发结论**：在复杂逻辑下，读懂并修正 AI 错误的时间远超直接编写代码的时间。
**视觉建议**: 折线图展示错误数量的“爆炸”趋势；使用“断裂的齿轮”图标象征逻辑崩溃。

**第4页**
**标题**: 总结：复杂度决定 AI Coding 的存亡价值
**副标题**: SAP ABAP AI 效率与质量对比验证总览
**内容要点**:
1. **核心规律**：提效随复杂度增加而骤降（简单 +50% → 中等 0% → 复杂 -60%）。
2. **四大瓶颈**：数据字典幻觉、接口规则盲区、修复成本反超、提示词遵循度低。
3. **工具选型**：Claude Code 在文档解析与业务理解上整体优于 GitHub Copilot。
4. **最终结论**：AI 目前仅适用于简单报表辅助，尚未具备处理 SAP 中高难度开发的能力。
**视觉建议**: 四象限矩阵图（复杂度 vs 提效百分比）；总结性金字塔模型。
